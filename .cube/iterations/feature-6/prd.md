# PRD：迭代 6 — 导演逻辑、邀请与收束

## 1. 功能概述

本次迭代在当前 `/discussion/[sessionId]` 会话详情页内实现 Director 导演逻辑，让系统从“角色机械轮流发言”升级为“由主持人主动控场的讨论节奏”。系统需要根据当前 session 状态、消息历史和用户意图，判断下一步是继续角色发言、邀请用户参与、提示事件候选，还是收束并输出总结。

本迭代包含以下功能点：

- Director 决策：输出 `continue`、`invite_user`、`trigger_event`、`conclude` 四类动作。
- 主持人控场：支持开场、转场、邀请、事件候选解释和总结消息样式。
- 用户邀请：在讨论详情页展示邀请卡，用户可回应或跳过，讨论继续围绕用户输入推进。
- 收束总结：用户主动要求总结或系统达到收束条件时，由主持人生成结构化总结，并在成功后完成会话。
- 讨论操作：更多操作中的“总结当前结论”真实触发总结流程。
- 会话入口一致性：能力全部绑定当前 session，不新增“讨论”一级 Tab 依赖；从首页、最近讨论或会话页恢复后均可进入当前讨论详情页。

## 2. Functional Requirements

### FR-001 Director 决策动作

- **优先级**：P0
- **功能点描述**：系统必须提供独立 Director 决策能力，基于当前 session state、message history、最近用户意图和最近调度结果，输出下一步动作。
- **输入**：当前 session、消息历史、触发来源、最近消息、可选用户意图结果。
- **输出**：结构化决策，动作必须为 `continue`、`invite_user`、`trigger_event`、`conclude` 之一，并包含原因、置信度、阶段变化建议、调度提示和可选事件候选。
- **异常处理**：当系统无法产出有效决策时，用户应看到“暂时无法判断下一步”的错误反馈，并可重试或继续普通发言；不得破坏现有消息流和 session 状态。

### FR-002 继续讨论

- **优先级**：P0
- **功能点描述**：当讨论仍处于开场或展开阶段，且角色观点尚未充分覆盖时，Director 应决定继续推进角色发言。
- **输入**：当前讨论阶段、消息数量、最近角色发言、可参与角色列表。
- **输出**：继续讨论动作，以及下一位建议发言角色或发言方向提示。
- **异常处理**：当没有可调度角色时，用户应看到生成失败反馈，并允许主持人兜底回应或稍后重试。

### FR-003 邀请用户参与

- **优先级**：P0
- **功能点描述**：当讨论出现明显分歧、角色质疑用户、讨论分叉或接近收束时，系统应由主持人邀请用户参与当前讨论。
- **输入**：当前 session 的分歧信号、分叉方案、阶段信息、最近消息。
- **输出**：主持人邀请消息、待处理邀请记录、InviteCard 展示内容、输入区高亮提示。
- **异常处理**：如果已有 pending 邀请，系统不得重复创建新邀请；如果邀请创建失败，用户应看到明确错误反馈且讨论状态保持不变。

### FR-004 回应邀请

- **优先级**：P0
- **功能点描述**：用户可对当前 session 内的待处理邀请进行文本回应，回应后讨论围绕用户输入继续推进。
- **输入**：邀请 ID、用户回应内容、客户端消息 ID。
- **输出**：邀请状态变为已回应、用户消息进入消息流、Director 返回后续推进方向。
- **异常处理**：空内容不得提交为回应；邀请不存在、已失效或不属于当前 session 时，应隐藏失效 InviteCard 并提示邀请已失效；重复提交不得创建重复消息。

### FR-005 跳过邀请

- **优先级**：P0
- **功能点描述**：用户可跳过当前邀请，跳过后讨论继续推进，不阻塞后续角色发言。
- **输入**：邀请 ID、跳过动作、客户端消息 ID。
- **输出**：邀请状态变为已跳过，消息流可继续生成后续角色或主持人消息。
- **异常处理**：重复跳过同一邀请不得重复更新状态；跳过失败时，InviteCard 保持可操作并展示错误反馈。

### FR-006 邀请频率控制

- **优先级**：P0
- **功能点描述**：系统必须避免过度邀请用户，保证邀请不会频繁打断讨论。
- **输入**：当前 session 最近邀请记录、消息数量、pending 邀请状态、邀请总次数。
- **输出**：满足节流规则时才允许新建邀请；未满足时应选择继续讨论或其他非打断动作。
- **异常处理**：pending 邀请未处理时不得新建邀请；冷却窗口内不得重复邀请。

### FR-007 主持人消息样式

- **优先级**：P1
- **功能点描述**：主持人消息需支持开场、转场、邀请、事件候选解释、阶段总结和最终总结的可辨识展示。
- **输入**：主持人消息类型、消息内容、阶段信息、可选 summary metadata。
- **输出**：消息流中展示符合语义的主持人消息样式，并在角色栏体现主持人控场和当前发言人。
- **异常处理**：当消息类型未知时，仍应以普通主持人消息展示，不应导致消息流渲染失败。

### FR-008 阶段变化呈现

- **优先级**：P1
- **功能点描述**：讨论阶段变化通过主持人转场/总结消息或轻量系统提示体现，不新增独立阶段评估面板。
- **输入**：Director 决策中的阶段变化建议、状态机结果。
- **输出**：用户能在消息流中理解讨论从开场、展开、冲突到收束的变化。
- **异常处理**：非法阶段变化不得更新 UI 状态，应保留原阶段并展示错误反馈。

### FR-009 事件候选提示

- **优先级**：P1
- **功能点描述**：当讨论出现明显正反阵营或适合后续事件的场景时，Director 可输出 `trigger_event`，但本迭代只展示事件候选解释。
- **输入**：分歧消息、候选事件类型、事件候选原因。
- **输出**：主持人解释“这里已形成某类候选事件”的消息，或轻量系统提示。
- **异常处理**：不得在本迭代创建真实 EventRecord、投票接口或完整事件交互；事件候选生成失败时应退化为继续讨论或主持人解释。

### FR-010 用户主动总结

- **优先级**：P0
- **功能点描述**：当用户通过输入或更多操作选择“总结当前结论”时，系统应触发 Director 收束判断，并由主持人生成当前结论总结。
- **输入**：用户总结/结束信号、当前 session、消息历史、更多操作入口触发。
- **输出**：主持人总结消息或最终总结卡，包含共识、分歧、建议、下一步等结构化内容。
- **异常处理**：总结生成失败时不得将 session 标记为 completed；用户应看到“总结生成失败”反馈，并可重试。

### FR-011 自动收束

- **优先级**：P0
- **功能点描述**：当讨论达到轮次上限、用户要求总结、角色达成共识或连续同意且无明显新信息时，Director 应输出收束动作。
- **输入**：轮次数、用户结束信号、角色发言一致性、消息历史。
- **输出**：主持人最终总结消息，成功后会话状态进入 completed。
- **异常处理**：必须先成功生成总结消息，再推进到 closing 并完成会话；状态更新失败时不得丢失总结消息。

### FR-012 总结后操作入口

- **优先级**：P1
- **功能点描述**：主持人总结完成后，页面应提供返回首页、查看会话页、继续追问当前 session 的入口。
- **输入**：已完成总结的 session、最终总结消息 ID、用户点击的后续操作。
- **输出**：返回首页时离开当前讨论；查看会话页时进入会话中心；继续追问时停留在当前讨论详情页并恢复为可继续输入状态。
- **异常处理**：继续追问应视为显式 resume，保留总结 checkpoint；如果恢复失败，应提示用户重试并保持当前总结可见。

### FR-013 会话页恢复后触发导演逻辑

- **优先级**：P0
- **功能点描述**：用户从会话页恢复进行中的 session 后，仍可触发邀请、总结和继续讨论流程。
- **输入**：会话页选择的 session、当前 session 状态和历史消息。
- **输出**：进入 `/discussion/[sessionId]` 后，Director 行为与从首页新建会话进入时一致。
- **异常处理**：会话不存在时，应提示会话不存在，并引导返回首页或会话页。

### FR-014 失败反馈

- **优先级**：P0
- **功能点描述**：Director 动作失败、邀请响应失败、阶段推进失败、总结失败时，用户必须获得明确可理解的反馈。
- **输入**：失败类型、失败消息、当前 session 状态。
- **输出**：错误提示、可重试入口或保留当前消息流。
- **异常处理**：任何失败不得造成重复消息、丢失消息、错误完成会话或跨 session 状态污染。

## 3. Non-Functional Requirements

### NFR-001 可测试性

Director 决策必须可预测地测试。自动化测试中应能使用 MockDirector 或等价机制覆盖 `continue`、`invite_user`、`conclude`，避免依赖真实 LLM 随机输出。

### NFR-002 幂等性

邀请回应和跳过操作必须幂等。同一个 invitation 已经 responded、skipped 或 expired 后，重复提交不得创建重复消息或重复推进讨论状态。

### NFR-003 状态一致性

Director 决策、邀请记录、消息流、讨论阶段和 session status 必须保持一致。总结成功前不得把 session 标记为 completed；pending 邀请不得跨 session 泄漏。

### NFR-004 用户体验

邀请和阶段变化应以轻量方式出现在现有讨论流中，不新增独立阶段评估面板，不破坏移动端原型中的消息流、角色栏、输入区和更多操作结构。

### NFR-005 安全与输入处理

用户邀请回应、快捷指令和总结触发均属于用户输入边界，展示到消息流前必须避免脚本注入；错误信息不得暴露内部堆栈、密钥或敏感调试信息。

### NFR-006 兼容性

本迭代必须兼容当前 Next.js App Router、客户端优先架构和 IndexedDB 本地持久化方向；不要求引入远端数据库或用户认证。

## 4. 验收标准

| 需求 | 可测试条件 |
|---|---|
| FR-001 | 给定不同 session 状态和消息历史时，Director 能输出 `continue`、`invite_user`、`trigger_event`、`conclude` 中的合法动作，并包含可展示原因。 |
| FR-002 | 当开场消息不足或角色观点尚未覆盖时，系统继续调度角色发言，而不是固定机械轮转。 |
| FR-003 | 当讨论出现分叉方案或明显分歧时，消息流出现主持人邀请消息，页面展示 InviteCard，输入区出现高亮提示。 |
| FR-004 | 用户填写邀请回应后，InviteCard 状态更新为已回应，用户消息进入消息流，后续讨论围绕用户回应继续。 |
| FR-005 | 用户选择跳过邀请后，InviteCard 不再阻塞讨论，后续角色或主持人消息可继续生成。 |
| FR-006 | 同一 session 存在 pending 邀请时再次触发 Director，不会创建第二个 pending 邀请；冷却窗口内不会重复邀请。 |
| FR-007 | 主持人开场、转场、邀请和总结消息在 UI 上能与普通角色消息区分，角色栏可体现当前发言人。 |
| FR-008 | 阶段变化通过消息流中的主持人转场/总结消息或轻量提示体现，页面不出现独立阶段评估面板。 |
| FR-009 | `trigger_event` 只展示事件候选解释，不创建真实投票交互、EventRecord 或投票提交能力。 |
| FR-010 | 从更多操作点击“总结当前结论”后，系统生成主持人总结；若失败，session 不进入 completed。 |
| FR-011 | 当达到轮次上限、用户要求总结或角色达成共识时，系统生成最终总结，成功后会话状态为 completed。 |
| FR-012 | 最终总结完成后，用户可返回首页、查看会话页或继续追问当前 session；继续追问保留总结 checkpoint 并恢复讨论。 |
| FR-013 | 从会话页进入进行中 session 后，可继续触发邀请和总结，行为与新建会话进入一致。 |
| FR-014 | Director 决策失败、非法阶段变化、邀请失效、重复邀请响应、总结失败等场景均有明确用户反馈，且不会造成消息或状态重复。 |
| NFR-001 | 单元测试覆盖 `continue`、`invite_user`、`conclude`，e2e 覆盖系统邀请用户、用户回应、会话页进入后触发邀请和总结。 |
| NFR-002 | 对同一邀请重复提交回应或跳过时，系统不创建重复用户消息。 |
| NFR-003 | 总结生成失败时，session 仍保持 running 或原状态；总结成功后再进入 completed。 |
| NFR-005 | 用户输入包含 HTML 或脚本片段时，消息流按文本安全展示。 |

## 5. Out of Scope

- 不实现完整爽点事件检测、真实 EventRecord、投票 API 或事件投票交互；这些属于迭代 7。
- 不实现复杂语义冲突判断或高级 Prompt 调优；本迭代以清晰可测试规则和可预测 Mock 为主。
- 不新增“讨论”一级 Tab 依赖；讨论页继续作为 `/discussion/[sessionId]` 会话详情页存在。
- 不引入远端数据库、用户认证、多端同步或权限系统。
- 不重写 AgentRuntime、Scheduler、IntentClassifier 或 StateMachine 的职责边界；Director 只负责节奏决策，不直接替代角色内容生成。
- 不修改 `docs/requirements/` 原始需求文档和原型文件。

## 依赖说明

- 依赖现有讨论页消息流、角色栏、输入区和更多操作结构。
- 依赖前序迭代已完成的 session 生命周期、StateMachine、Scheduler、IntentClassifier、用户介入和快捷命令能力。
- 依赖当前 API 统一响应格式：`{ success, data, error, requestId }`。
- 依赖本地持久化方向保存 session、消息、Director 决策和邀请记录。
