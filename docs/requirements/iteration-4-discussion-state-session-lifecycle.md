# 迭代 4：讨论状态机与会话生命周期

> 评审日期：2026-05-20  
> 评审结论：通过。根据导航架构修正，本迭代保留会话中心和状态机能力，但讨论页不再作为底部一级 Tab；会话页成为已有 session 进入讨论详情页的主入口。  
> 修订来源：底部导航架构调整评审。  
> 导航基线：底部一级导航为 `首页 / 会话 / 模板 / 设置`；讨论页保留为 `/discussion/[sessionId]` 会话详情页，不作为一级 Tab。

---

## 迭代目标

让讨论具备阶段推进，同时完善会话中心，支持查看、搜索、筛选、恢复、归档讨论。

本迭代将会话中心与讨论详情页打通：会话页负责管理和恢复 session，讨论页仅作为 `/discussion/[sessionId]` 的当前会话工作台。

## 原型映射

| 原型区域 | 本迭代要求 |
|---|---|
| 会话中心 | 全部、进行中、已完成、已归档筛选 |
| 会话搜索 | 可搜索标题、模板、角色 |
| 会话卡片 | 展示状态、模板、角色数、事件/消息数、时间 |
| 会话操作 | 新建、导出入口、归档、批量入口 |
| 讨论详情页 `/discussion/[sessionId]` | 通过有效 sessionId 加载消息、角色和状态；不作为底部一级 Tab |
| 最近讨论 | 首页最近讨论读取同一会话数据源，并携带 sessionId 进入讨论详情页 |

## 导航与入口约束

| 约束 | 说明 |
|---|---|
| 讨论页不是一级 Tab | 底部导航不展示“讨论”；讨论页只能作为具体 session 的详情页访问 |
| 必须携带 sessionId | `/discussion/[sessionId]` 是唯一讨论页路由形态，不允许无 sessionId 默认讨论 |
| 已有讨论主入口 | 会话页会话卡片是已有 session 的主要恢复入口 |
| 首页快捷入口 | 首页最近讨论可作为快捷入口，但必须读取同一 session 数据源 |
| 职责边界 | 会话页负责搜索、筛选、归档、状态展示；讨论页负责当前会话消息流和交互 |

## 状态机基线

```text
idle → opening → developing → climax → closing
```

### Step 4｜需求评审意见

#### 1. 完整性

| 问题 | 评审意见 | 修订建议 |
|---|---|---|
| 会话状态与讨论 phase 容易混用 | `status` 面向会话列表和生命周期，`phase` 面向讨论引擎推进，两者应关联但不等同 | 保留 BE-008：closing 后可标记 completed；实现时明确 status 与 phase 的转换规则 |
| 归档 / 完成 / 恢复需要幂等 | 用户可能重复点击归档、恢复，或刷新后重复提交状态更新 | `PATCH /api/sessions/:sessionId/status` 应支持幂等处理，并返回当前最终状态 |
| 首页最近讨论与会话页数据源需统一 | 若首页最近讨论仍用组件 Mock，会出现与会话中心不一致 | 保留 FE-008 和 Repository 查询要求，最近讨论必须复用 session 数据源 |
| 讨论页阶段展示已被删除 | 原型调整后，不应再在讨论页新增独立阶段评估组件 | 保留 FE-010 与验收标准，phase 可用于引擎和列表状态，不强制在讨论页展示 |
| 状态历史有调试价值 | 状态机错误很难从最终状态反推 | 保留 BE-010，记录 state history，便于调试和后续 Director 复盘 |

#### 2. 合理性

| 检查点 | 结论 |
|---|---|
| 技术可行性 | 可行。迭代 0 已预留 state-machine 和会话模块，迭代 1-3 已建立 session 创建、运行时和消息流基础。 |
| 产品边界 | 合理。会话页做跨会话管理，讨论详情页做当前 session 工作台，避免两个页面功能重复。 |
| API 范围 | 可控。新增 `GET /api/sessions`、`PATCH /api/sessions/:sessionId/status`、`GET /api/sessions/:sessionId/state` 即可覆盖本迭代核心。 |
| UI 复杂度 | 中等。搜索、筛选、归档和恢复需要统一 Store / service，同步首页最近讨论和会话页列表。 |

#### 3. 一致性

| 对象 | 结论 |
|---|---|
| 与迭代 0 | 对齐。迭代 0 已要求 `/sessions` 页面骨架、`/discussion/[sessionId]` 页面骨架、engine/state-machine 预留。 |
| 与迭代 1 | 对齐。迭代 1 创建 session 并进入 `/discussion/[sessionId]`，迭代 4 扩展 session 查询、恢复和归档。 |
| 与迭代 2 | 对齐。迭代 2 的 Orchestrator / Scheduler 可被状态机消费，不需要本迭代重写 Agent 运行时。 |
| 与迭代 3 | 对齐。迭代 3 负责消息流体验，迭代 4 不改变消息组件职责，只增加会话生命周期和 phase 推进。 |
| 与迭代 5-9 | 对齐。用户意图、Director、事件和导出均明确后置，避免范围膨胀。 |

#### 4. 风险点

| 风险 | 等级 | 影响 | 应对 |
|---|---|---|---|
| `status` 与 `phase` 定义混乱 | P0 | 会话列表、引擎推进和完成状态出现不一致 | 明确 `status` 是会话生命周期，`phase` 是讨论阶段；只通过 service 层转换 |
| UI 直接修改 session 状态 | P0 | 会话页、首页最近讨论和讨论页状态串扰 | 统一通过 SessionService / Repository 更新，不允许组件本地硬改最终状态 |
| 无 sessionId 进入讨论页 | P0 | 与导航基线冲突，恢复默认讨论页问题 | 保留 FE-009；无效 sessionId 应展示错误并引导回首页或会话页 |
| 归档后仍出现在进行中列表 | P1 | 用户感知列表不可信 | Repository 查询必须按 status 过滤，归档后刷新列表 |
| phase 推进规则过度复杂 | P1 | 状态机不可测、难以维护 | 本迭代只实现固定合法流转和基础触发规则，复杂 Director 判断后置 |
| 讨论页重新加入阶段评估面板 | P1 | 与会话页职责冲突，重复 UI | 验收时检查讨论页不提供会话列表 / 筛选 / 阶段评估入口 |

### Step 5｜最终需求修订摘要

| 变更项 | 最终需求 | 原因 |
|---|---|---|
| 讨论页定位 | 讨论页保留为 `/discussion/[sessionId]` 当前会话工作台，不作为 BottomNav 一级入口 | 避免讨论页与会话页重复，符合最新导航基线 |
| 会话页定位 | 会话页承担已有 session 的搜索、筛选、恢复、归档和状态展示 | 明确跨会话管理职责 |
| 阶段展示 | 不在讨论页新增独立阶段评估组件；phase 仅用于引擎推进、列表状态和调试 | 避免 UI 冗余和职责冲突 |
| 状态机 | 保留 `idle → opening → developing → climax → closing`，并校验非法转换 | 满足蓝图中讨论状态机要求 |
| 状态更新接口 | `PATCH /api/sessions/:sessionId/status` 支持 archive、complete、resume，并应具备幂等性 | 保证会话生命周期操作稳定可测 |
| 数据源 | 首页最近讨论、会话页列表、讨论详情恢复均使用统一 session 数据源 | 避免组件 Mock 数据不一致 |
