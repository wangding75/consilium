# 重新拆分后的迭代计划总览

## 拆分原则

1. 原型 5 个 Tab 必须全部进入迭代计划：`首页 / 讨论 / 会话 / 模板 / 设置`。
2. 迭代 0 不只初始化基础架构，还必须生成完整模块骨架代码。
3. 每个后续迭代必须沿着迭代 0 的模块骨架继续完善。
4. 每个迭代同时明确前端需求、后端/API需求、验收标准。
5. 多 Agent 架构、状态机、导演逻辑、爽点机制仍是产品核心。
6. 模板、设置、Provider 配置不再完全排除，但放在核心讨论链路之后实现。

## 迭代总表

| 迭代 | 名称 | 主要页面 | 核心目标 |
|---|---|---|---|
| 迭代 0 | 项目脚手架与模块骨架 | 全局骨架 | 建立前端、后端/API、引擎、领域模型、Mock 数据的代码边界 |
| 迭代 1 | 首页与创建讨论会话 | 首页 | 用户能输入议题、选择默认模板和模型策略，创建讨论会话 |
| 迭代 2 | 多 Agent 运行时基础架构 | 讨论 | 建立 Host / Expert / Critic 等 Agent 抽象和基础串行调度 |
| 迭代 3 | 讨论页消息流联动 | 讨论 | 讨论页完成真实数据渲染、消息发送、角色发言展示 |
| 迭代 4 | 状态机与会话生命周期 | 会话 + 讨论 | 讨论阶段推进，会话可查询、筛选、恢复、归档 |
| 迭代 5 | 用户介入与快捷指令 | 讨论 | 用户可插话、指挥、定夺，系统识别意图并影响调度 |
| 迭代 6 | 导演逻辑、邀请与收束 | 讨论 | 系统主动邀请用户、判断分歧、收束总结 |
| 迭代 7 | 爽点事件与投票机制 | 讨论 | 实现打脸、站队、投票、反转事件及事件卡渲染 |
| 迭代 8 | 模板、角色与模型策略 | 模板 + 首页 | 模板详情、角色列表、事件规则、节奏配置、模型策略可用 |
| 迭代 9 | 设置、Provider 与数据闭环 | 设置 + 会话 | Provider/模型配置、Prompt 配置、本地数据、导出闭环 |

## 前后端边界

早期可采用 Next.js 全栈形态：

| 层 | 建议实现 |
|---|---|
| 前端 | Next.js App Router + React Components |
| 后端/API | Next.js Route Handlers / Server Actions |
| 引擎层 | `src/engine` 下的纯 TypeScript 模块 |
| 运行时服务 | `src/server/services` 下封装业务服务 |
| 存储 | Phase 1 可先 IndexedDB + Mock Repository，后续替换服务端 DB |
| LLM | 统一通过 `src/llm/client.ts` 和 server-side provider adapter 调用 |

虽然早期不一定有独立后端服务，但每个迭代都必须有清晰 API/服务边界，避免 UI 直接写业务逻辑。
