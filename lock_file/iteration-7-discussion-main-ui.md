# 迭代 7：讨论主界面

## 1. 迭代目标

在讨论引擎能力基本完成后，构建正式的讨论主界面，让用户能够以清晰、连续、可交互的方式体验多 Agent 讨论。

本迭代重点是 UI 承接已有 engine 能力：

```text
消息流 + 角色发言 + 当前发言人 + 用户输入 + 邀请用户 + 事件展示
```

本迭代不重新设计引擎规则，只负责把迭代 1-6 的能力产品化呈现出来。

---

## 2. 迭代范围

### 2.1 本迭代包含

| 范围 | 说明 |
|---|---|
| ChatContainer | 讨论页主容器 |
| MessageBubble | 展示用户、主持人、角色、系统消息 |
| EventCard | 展示打脸、站队、投票、反转事件 |
| InviteCard | 展示系统邀请用户参与 |
| InputArea | 用户输入区，支持基础输入模式 |
| RosterBar | 展示参与讨论的角色和当前发言人 |
| TypingIndicator | 展示 AI 正在生成状态 |
| TopBar | 展示模板名、议题、讨论状态 |
| Engine 连接 | UI 通过 store/engine 触发讨论流程 |

### 2.2 本迭代不包含

| 不包含内容 | 原因 |
|---|---|
| 模板选择 UI | 当前仍通过默认模板或简单入口进入 |
| LLM 配置 UI | 当前不纳入 |
| 角色配置 UI | 当前不纳入 |
| 移动端深度适配 | 非功能模块，后续产品化处理 |
| 复杂动画和动效 | 可适度实现，不作为核心验收 |
| 埋点和监控 | 非功能模块，后续处理 |

## 3. 产品需求

### PRD-7.1：用户能清晰阅读讨论消息

**需求说明**  
作为用户，我需要清楚看到每条消息是谁说的、属于什么类型、在讨论中处于什么位置。

| 编号 | 需求 |
|---|---|
| PRD-7.1.1 | 用户消息、主持人消息、角色消息、系统消息视觉上可区分 |
| PRD-7.1.2 | 每条角色消息显示角色名称 |
| PRD-7.1.3 | 消息按时间顺序展示 |
| PRD-7.1.4 | 生成中的消息需要有 loading 或 streaming 状态 |
| PRD-7.1.5 | 长消息应保持可读，不应破坏布局 |

---

### PRD-7.2：用户能看到事件发生

**需求说明**  
作为用户，我希望打脸、站队、投票、反转等关键事件能被突出展示，而不是淹没在普通文本中。

| 编号 | 需求 |
|---|---|
| PRD-7.2.1 | 打脸事件以事件卡展示 |
| PRD-7.2.2 | 站队事件以事件卡展示，并显示阵营信息 |
| PRD-7.2.3 | 投票事件以事件卡展示，并显示投票结果或投票动作 |
| PRD-7.2.4 | 反转事件以事件卡展示，并显示新增条件或突发信息 |
| PRD-7.2.5 | 事件卡在消息流中按发生顺序展示 |

---

### PRD-7.3：用户能响应系统邀请

**需求说明**  
作为用户，当系统邀请我参与讨论时，我需要能直接回应或跳过。

| 编号 | 需求 |
|---|---|
| PRD-7.3.1 | 系统邀请用户时展示 InviteCard |
| PRD-7.3.2 | InviteCard 显示邀请问题或提示 |
| PRD-7.3.3 | 用户可以输入回应 |
| PRD-7.3.4 | 用户可以跳过邀请，让讨论继续 |
| PRD-7.3.5 | 用户回应后，邀请状态应清除 |

---

### PRD-7.4：用户能持续参与讨论

**需求说明**  
作为用户，我需要一个稳定的输入区，支持继续插话、指挥和定夺。

| 编号 | 需求 |
|---|---|
| PRD-7.4.1 | 页面底部提供固定或稳定可见的输入区 |
| PRD-7.4.2 | 用户输入后消息进入讨论流 |
| PRD-7.4.3 | 输入区在 AI 生成时应防止重复提交 |
| PRD-7.4.4 | 输入区可展示基础模式提示，例如“插话 / 指挥 / 定夺” |
| PRD-7.4.5 | 支持快捷指令入口，例如“总结一下”“继续讨论” |

---

### PRD-7.5：用户能看到当前讨论角色

**需求说明**  
作为用户，我希望知道当前有哪些角色参与讨论，以及现在谁在发言。

| 编号 | 需求 |
|---|---|
| PRD-7.5.1 | 页面展示当前模板角色列表 |
| PRD-7.5.2 | 主持人和普通角色应有区别 |
| PRD-7.5.3 | 当前发言角色需要高亮 |
| PRD-7.5.4 | 用户自身也可以作为参与者展示 |

## 4. 技术需求

### TECH-7.1：组件结构

建议目录：

```text
src/components/chat/
├── ChatContainer.tsx
├── MessageBubble.tsx
├── EventCard.tsx
├── InviteCard.tsx
├── InputArea.tsx
├── RosterBar.tsx
├── TypingIndicator.tsx
└── TopBar.tsx
```

| 编号 | 技术需求 |
|---|---|
| TECH-7.1.1 | Chat UI 组件集中放在 `components/chat` |
| TECH-7.1.2 | 组件接收标准 props，不直接读取过多全局状态 |
| TECH-7.1.3 | ChatContainer 负责组合组件和连接 store |
| TECH-7.1.4 | 组件样式使用 Tailwind 和 CSS Variables |

---

### TECH-7.2：MessageBubble

| 编号 | 技术需求 |
|---|---|
| TECH-7.2.1 | 支持 host、character、user、system、event 类型消息 |
| TECH-7.2.2 | 支持 isStreaming 状态展示 |
| TECH-7.2.3 | 支持 character 信息传入，包括名称、颜色、是否主持人 |
| TECH-7.2.4 | 不同消息类型视觉区分清晰 |
| TECH-7.2.5 | 消息内容需要安全渲染，默认不执行 HTML |

---

### TECH-7.3：EventCard

| 编号 | 技术需求 |
|---|---|
| TECH-7.3.1 | EventCard 接收 DiscussionEvent |
| TECH-7.3.2 | 根据 event.type 渲染不同标题和样式 |
| TECH-7.3.3 | 支持显示 trigger 信息、参与角色、metadata |
| TECH-7.3.4 | EventCard 不负责事件检测，只负责展示 |

---

### TECH-7.4：InviteCard

| 编号 | 技术需求 |
|---|---|
| TECH-7.4.1 | InviteCard 显示邀请文案和操作入口 |
| TECH-7.4.2 | 支持回应和跳过 |
| TECH-7.4.3 | 回应后调用统一用户消息处理流程 |
| TECH-7.4.4 | 跳过后调用 engine 继续讨论 |

---

### TECH-7.5：InputArea

| 编号 | 技术需求 |
|---|---|
| TECH-7.5.1 | 支持输入文本并提交 |
| TECH-7.5.2 | 支持 Enter / Shift+Enter 基础交互 |
| TECH-7.5.3 | 支持 isGenerating 时禁用重复提交 |
| TECH-7.5.4 | 支持快捷指令 chips |
| TECH-7.5.5 | 可预留 mode：interrupt、direct、decide |

---

### TECH-7.6：Store 与 Engine 集成

| 编号 | 技术需求 |
|---|---|
| TECH-7.6.1 | ChatContainer 通过 chat store 获取 messages、phase、isGenerating、currentSpeaker |
| TECH-7.6.2 | 用户提交输入时调用 engine handler，而非直接拼接 AI 消息 |
| TECH-7.6.3 | AI 生成状态需要正确更新 isGenerating |
| TECH-7.6.4 | 事件、邀请、普通消息统一进入消息流或对应状态 |
| TECH-7.6.5 | 页面层不实现 Director、Scheduler、Events 业务规则 |

---

### TECH-7.7：自动滚动与空状态

| 编号 | 技术需求 |
|---|---|
| TECH-7.7.1 | 新消息出现时自动滚动到底部 |
| TECH-7.7.2 | 用户手动向上查看历史时避免强制打断，可先简化处理 |
| TECH-7.7.3 | 无消息时展示空状态和开始讨论提示 |
| TECH-7.7.4 | 生成中展示 TypingIndicator |

## 7. 需求评审关注点

进入本迭代开发前，需求评审需要确认：

| 评审项 | 需要确认的问题 |
|---|---|
| 消息样式 | 主持人、角色、用户消息是否采用不同布局 |
| 事件表现 | 事件卡是否需要明显颜色区分 |
| 输入模式 | 当前是否实现插话/指挥/定夺 tabs，还是只作为提示 |
| 邀请交互 | InviteCard 是弹出、卡片，还是消息流内展示 |
| 当前角色 | 是否需要角色头像、颜色、状态 |
| 最小 UI 粒度 | 本迭代追求可用还是精致 |

---

## 8. 迭代完成定义

满足以下条件时，迭代 7 可视为完成：

1. 用户能通过正式讨论主界面完成一次讨论。
2. 普通消息、事件、邀请、生成状态均可展示。
3. 用户可以在讨论过程中继续输入。
4. 当前发言人和角色列表可见。
5. UI 层与 engine 业务逻辑解耦。
6. 工程构建和 Lint 通过。
