# 迭代 6：爽点机制

## 1. 迭代目标

在多 Agent 讨论、状态机、用户介入、导演逻辑基础上，为讨论增加“戏剧性”和“可看性”。

本迭代实现四类爽点事件：

```text
打脸 face_slap：后续角色用事实或逻辑推翻前一角色断言
站队 camp：角色围绕争议议题形成阵营
投票 vote：争论无法收束时发起表态
反转 reverse：引入突发信息或极端假设，改变讨论方向
```

目标不是做复杂语义推理，而是 Phase 1 先用关键词和规则让事件机制跑通，为 Phase 2 语义检测升级预留接口。

---

## 2. 迭代范围

### 2.1 本迭代包含

| 范围 | 说明 |
|---|---|
| Event 类型 | 定义事件数据结构和事件消息类型 |
| 打脸检测 | 基于断言和反驳关键词触发 |
| 站队检测 | 基于支持/反对倾向形成阵营 |
| 投票检测 | 争论多轮无结论时触发投票 |
| 反转检测 | 基于突发信息关键词或导演主动触发 |
| 事件去重 | 每类事件在单次讨论中限制触发次数 |
| Director 集成 | Director 可以调用事件检测或消费事件结果 |
| Prompt 注入 | 事件发生后影响后续角色发言 |

### 2.2 本迭代不包含

| 不包含内容 | 原因 |
|---|---|
| 语义冲突检测 | Phase 2 使用 Embedding 或 LLM Judge 升级 |
| 精美事件卡 UI | 迭代 7 负责展示 |
| 高级投票图表 | 不在当前阶段 |
| 用户自定义事件规则 | 不在当前阶段 |
| 事件运营后台 | 后续产品化再考虑 |

## 3. 产品需求

### PRD-6.1：支持打脸事件

**需求说明**  
作为用户，我希望看到角色之间出现有张力的观点推翻，而不是温和重复。

| 编号 | 需求 |
|---|---|
| PRD-6.1.1 | 当后续角色明显反驳前一角色断言时，系统可触发打脸事件 |
| PRD-6.1.2 | 打脸事件需要记录攻击方、被反驳方和触发原因 |
| PRD-6.1.3 | 打脸事件不应过于频繁，避免变成噱头 |
| PRD-6.1.4 | 事件触发后，后续讨论应围绕冲突点继续推进 |

---

### PRD-6.2：支持站队事件

**需求说明**  
作为用户，我希望当讨论出现重大方向分歧时，角色能自然形成阵营。

| 编号 | 需求 |
|---|---|
| PRD-6.2.1 | 系统能识别支持方和反对方 |
| PRD-6.2.2 | 当双方角色数量达到条件时触发站队事件 |
| PRD-6.2.3 | 站队事件需要标记每个阵营角色 |
| PRD-6.2.4 | 站队后，主持人可以提示用户关注核心分歧 |

---

### PRD-6.3：支持投票事件

**需求说明**  
作为用户，我希望争论无法收束时，系统能让角色投票，快速形成局势判断。

| 编号 | 需求 |
|---|---|
| PRD-6.3.1 | 当同一议题争论多轮没有结论时，可触发投票 |
| PRD-6.3.2 | 投票事件需要记录每个角色的选择 |
| PRD-6.3.3 | 投票后主持人应总结投票结果 |
| PRD-6.3.4 | 投票不应在讨论刚开始时触发 |

---

### PRD-6.4：支持反转事件

**需求说明**  
作为用户，我希望讨论中偶尔出现突发信息或极端假设，让讨论更有变化。

| 编号 | 需求 |
|---|---|
| PRD-6.4.1 | 系统可基于“探子来报、忽然、不曾想、意外”等触发反转 |
| PRD-6.4.2 | 反转事件需要说明新增条件或突发信息 |
| PRD-6.4.3 | 反转后角色应重新评估原观点 |
| PRD-6.4.4 | 单次讨论中反转次数应严格限制 |

## 4. 技术需求

### TECH-6.1：事件类型定义

建议定义：

```ts
export type EventType = 'face_slap' | 'camp' | 'vote' | 'reverse'

export interface DiscussionEvent {
  id: string
  type: EventType
  title: string
  description: string
  triggerMessageIds: string[]
  participantRoleIds: string[]
  createdAt: number
  metadata?: Record<string, unknown>
}
```

| 编号 | 技术需求 |
|---|---|
| TECH-6.1.1 | Event 类型应独立于 UI 组件定义 |
| TECH-6.1.2 | 事件可转换为 system/event message |
| TECH-6.1.3 | 事件 metadata 支持扩展阵营、投票结果等信息 |
| TECH-6.1.4 | EngineState 应记录已触发事件，避免重复触发 |

---

### TECH-6.2：events.ts 模块

建议新增：

```text
src/engine/events.ts
```

核心函数：

```ts
export function detectEvents(state: EngineState): DiscussionEvent[]
```

| 编号 | 技术需求 |
|---|---|
| TECH-6.2.1 | 所有事件检测逻辑集中在 `events.ts` |
| TECH-6.2.2 | 事件检测输入为 EngineState |
| TECH-6.2.3 | 事件检测输出为 DiscussionEvent 数组 |
| TECH-6.2.4 | 单次 engine step 中最多触发一个主事件，避免信息过载 |
| TECH-6.2.5 | 事件检测规则应可单元测试 |

---

### TECH-6.3：打脸检测规则

建议规则：

| 规则 | 示例关键词 |
|---|---|
| 前文存在强断言 | 应该、必须、一定、肯定、必然、确是 |
| 后文存在反驳 | 但是、不对、此言差矣、错了、事实并非如此 |
| 角色不同 | 反驳者不能是被反驳者本人 |
| 未触发过 | 当前讨论中 face_slap 未达到上限 |

| 编号 | 技术需求 |
|---|---|
| TECH-6.3.1 | 实现 `detectFaceSlap(state)` |
| TECH-6.3.2 | 需要返回攻击方、被反驳方、相关消息 |
| TECH-6.3.3 | 默认单次讨论最多触发 1-2 次 |

---

### TECH-6.4：站队检测规则

建议规则：

| 阵营 | 示例关键词 |
|---|---|
| 支持方 | 支持、赞成、应该、可行、同意 |
| 反对方 | 反对、不行、不可、不应、不同意 |

触发条件：

```text
支持方角色数 >= 2 且反对方角色数 >= 1
或
支持方角色数 >= 1 且反对方角色数 >= 2
```

| 编号 | 技术需求 |
|---|---|
| TECH-6.4.1 | 实现 `detectCamp(state)` |
| TECH-6.4.2 | 需要输出 proCamp 和 conCamp |
| TECH-6.4.3 | 站队事件默认单次讨论最多触发 1 次 |

---

### TECH-6.5：投票检测规则

建议规则：

| 条件 | 说明 |
|---|---|
| 讨论已进行多轮 | currentTurn >= 3 或模板配置阈值 |
| 存在分歧 | 最近消息包含争议关键词 |
| 无结论 | 最近消息没有“结论是、总之、所以、因此”等收束表达 |
| 未触发过 | 当前讨论 vote 未触发 |

| 编号 | 技术需求 |
|---|---|
| TECH-6.5.1 | 实现 `detectVote(state)` |
| TECH-6.5.2 | 投票事件可先生成事件指令，由后续角色输出投票 |
| TECH-6.5.3 | 投票结果需要记录在 event metadata 中 |

---

### TECH-6.6：反转检测规则

建议规则：

| 触发来源 | 说明 |
|---|---|
| 关键词触发 | 探子来报、忽然、不曾想、意外、叛变、假设 |
| Director 主动触发 | 讨论过于平淡或进入高潮阶段时 |
| 模板配置 | 某些模板允许更强戏剧性 |

| 编号 | 技术需求 |
|---|---|
| TECH-6.6.1 | 实现 `detectReverse(state)` |
| TECH-6.6.2 | 反转事件应生成新增条件或场景变化描述 |
| TECH-6.6.3 | 默认单次讨论最多触发 1 次 |
| TECH-6.6.4 | 反转内容不能完全脱离用户议题 |

---

### TECH-6.7：Director 与 Prompt 集成

| 编号 | 技术需求 |
|---|---|
| TECH-6.7.1 | Director 可以返回 trigger_event 决策 |
| TECH-6.7.2 | Orchestrator 处理事件后，将事件写入 messageHistory |
| TECH-6.7.3 | Context Builder 在事件后向 Agent Prompt 注入事件信息 |
| TECH-6.7.4 | 事件触发后，后续角色需要回应事件影响 |

## 7. 需求评审关注点

进入本迭代开发前，需求评审需要确认：

| 评审项 | 需要确认的问题 |
|---|---|
| 事件名称 | 是否使用“打脸/站队/投票/反转”作为正式产品名 |
| 触发频率 | 每类事件单次讨论最多触发几次 |
| 戏剧化程度 | 反转事件允许多强的剧情化表达 |
| 投票形式 | 投票是角色自动投票，还是邀请用户发起 |
| 事件展示 | 当前是否只写入消息流，UI 卡片留到迭代 7 |
| 规则边界 | Phase 1 是否确认只做关键词规则，不做语义裁判 |

---

## 8. 迭代完成定义

满足以下条件时，迭代 6 可视为完成：

1. 四类事件有统一类型和检测入口。
2. 打脸、站队、投票、反转能在规则条件下触发。
3. 事件不会无限重复触发。
4. 事件能写入讨论上下文，并影响后续发言。
5. Director 和 Orchestrator 能处理事件结果。
6. 关键事件规则有测试覆盖。
7. 工程构建和 Lint 通过。
