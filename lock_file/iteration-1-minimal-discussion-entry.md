# 迭代 1：最小讨论入口

## 1. 迭代目标

在不引入配置 UI 的前提下，基于迭代 0 的项目脚手架，完成一个最小可运行的讨论入口：用户进入固定场景，输入一个议题，系统基于内置模板生成第一轮 AI 讨论内容。

本迭代目标不是实现完整多 Agent 调度，也不是实现精细 UI，而是验证产品最基础闭环：

```text
用户输入议题 -> 系统读取默认模板 -> 调用 LLM -> 生成讨论开场/第一轮内容 -> 页面展示结果
```

---

## 2. 迭代范围

### 2.1 本迭代包含

| 范围 | 说明 |
|---|---|
| 默认模板 | 内置一个默认模板，建议使用“三国军师团”或临时通用模板 |
| 议题输入 | 用户可以在讨论页输入本次讨论主题 |
| 开始讨论 | 用户点击开始后进入讨论流程 |
| 基础消息展示 | 页面能够展示用户议题、主持人开场、角色第一轮观点 |
| LLM 调用接入 | 能够通过统一 LLM Client 调用模型，支持 mock fallback |
| 基础 Prompt 拼接 | 根据模板、角色、用户议题生成最小可用 Prompt |
| 会话初始化 | 初始化当前会话的 sessionId、topic、messages |

### 2.2 本迭代不包含

| 不包含内容 | 原因 |
|---|---|
| 模板选择 UI | 当前只使用内置默认模板 |
| LLM 配置 UI | 当前通过环境变量或代码默认配置 |
| 角色配置 UI | 当前使用模板内置角色 |
| 完整多 Agent 调度 | 属于迭代 2 |
| 状态机阶段切换 | 属于迭代 3 |
| 用户意图识别 | 属于迭代 4 |
| 导演逻辑 | 属于迭代 5 |
| 爽点事件 | 属于迭代 6 |
| 完整 Chat UI | 属于迭代 7 |

## 3. 产品需求

### PRD-1.1：用户可以输入讨论议题

**需求说明**  
作为用户，我需要在讨论页输入一个议题，让 AI 角色围绕这个议题开始讨论。

| 编号 | 需求 |
|---|---|
| PRD-1.1.1 | `/chat` 页面提供议题输入框 |
| PRD-1.1.2 | 输入框支持用户输入不少于 1 个字符的文本 |
| PRD-1.1.3 | 议题为空时，开始按钮不可用或提示用户输入议题 |
| PRD-1.1.4 | 用户点击开始后，输入议题作为当前会话 topic |

---

### PRD-1.2：系统使用默认模板启动讨论

**需求说明**  
作为用户，我不需要先配置复杂信息，系统应该直接使用默认模板启动讨论。

| 编号 | 需求 |
|---|---|
| PRD-1.2.1 | 系统内置一个默认讨论模板 |
| PRD-1.2.2 | 默认模板至少包含 1 个主持人和 2 个讨论角色 |
| PRD-1.2.3 | 每个角色至少包含名称、人设、立场或能力描述、systemPrompt |
| PRD-1.2.4 | 启动讨论时自动加载默认模板 |

---

### PRD-1.3：系统生成第一轮讨论内容

**需求说明**  
作为用户，我输入议题后，需要看到 AI 已经开始围绕议题进行讨论。

| 编号 | 需求 |
|---|---|
| PRD-1.3.1 | 系统生成主持人开场内容 |
| PRD-1.3.2 | 系统生成至少 2 个角色的第一轮观点 |
| PRD-1.3.3 | 每条消息能区分用户、主持人、角色 |
| PRD-1.3.4 | 角色发言需要围绕用户议题，不应脱离主题 |
| PRD-1.3.5 | 若 LLM 调用失败，应展示失败提示或 fallback mock 内容 |

---

### PRD-1.4：基础讨论结果可读

**需求说明**  
作为用户，我需要能清楚看到谁说了什么，而不是只看到一段混合文本。

| 编号 | 需求 |
|---|---|
| PRD-1.4.1 | 用户议题作为用户消息展示 |
| PRD-1.4.2 | 主持人消息显示角色名和内容 |
| PRD-1.4.3 | 普通角色消息显示角色名和内容 |
| PRD-1.4.4 | 消息按生成顺序展示 |
| PRD-1.4.5 | 页面上能重新输入议题开始一次新的最小讨论 |

---

## 4. 技术需求

### TECH-1.1：默认模板数据

| 编号 | 技术需求 |
|---|---|
| TECH-1.1.1 | 在 `src/data/templates/default.ts` 定义默认模板 |
| TECH-1.1.2 | 模板结构应符合 `Template` 类型 |
| TECH-1.1.3 | 模板中至少包含 host role 和 2 个 character role |
| TECH-1.1.4 | 每个 role 必须包含 `id`、`name`、`isHost`、`personality`、`systemPrompt` |
| TECH-1.1.5 | 默认模板应可被后续模板注册表复用，不要写死在页面组件中 |

---

### TECH-1.2：LLM Client 最小实现

| 编号 | 技术需求 |
|---|---|
| TECH-1.2.1 | 在 `src/llm/client.ts` 实现统一 `LLMClient` 调用入口 |
| TECH-1.2.2 | 支持从环境变量读取默认模型配置 |
| TECH-1.2.3 | 至少实现 `chat(messages): Promise<string>` |
| TECH-1.2.4 | 支持 mock client，用于无 API Key 或模型调用失败时本地联调 |
| TECH-1.2.5 | 页面层不能直接依赖 OpenAI、Gemini、DeepSeek 等具体 SDK |

---

### TECH-1.3：Prompt Builder

| 编号 | 技术需求 |
|---|---|
| TECH-1.3.1 | 新增 `src/llm/prompt-builder.ts` 或 `src/engine/prompt.ts` |
| TECH-1.3.2 | 支持根据 template、role、topic 生成角色发言 Prompt |
| TECH-1.3.3 | Prompt 中应包含角色身份、讨论议题、输出要求 |
| TECH-1.3.4 | 第一轮发言应限制字数，避免输出过长 |
| TECH-1.3.5 | Prompt 构造逻辑不能散落在页面组件中 |

---

### TECH-1.4：会话初始化

| 编号 | 技术需求 |
|---|---|
| TECH-1.4.1 | 在 chat store 中维护当前 sessionId、topic、messages、isGenerating |
| TECH-1.4.2 | 点击开始讨论时创建新的 sessionId |
| TECH-1.4.3 | 将用户议题写入 messages |
| TECH-1.4.4 | 将 AI 生成结果按 Message 类型写入 messages |
| TECH-1.4.5 | 当前阶段可以不写入 IndexedDB，但状态结构要为后续存储预留 |

---

### TECH-1.5：最小讨论执行流程

建议在 `src/engine/start-discussion.ts` 中实现最小流程：

```text
startDiscussion(topic)
  -> load default template
  -> create session state
  -> append user message
  -> generate host opening
  -> generate first role opinions
  -> append AI messages
  -> return messages
```

| 编号 | 技术需求 |
|---|---|
| TECH-1.5.1 | 执行流程应封装在 engine 或 service 层，不写在 React 组件中 |
| TECH-1.5.2 | 至少支持串行生成主持人和角色消息 |
| TECH-1.5.3 | LLM 调用失败时不应导致页面崩溃 |
| TECH-1.5.4 | 每条消息都应包含 id、role、content、timestamp |

## 7. 需求评审关注点

进入本迭代开发前，需求评审需要确认：

| 评审项 | 需要确认的问题 |
|---|---|
| 默认模板 | 是否使用“三国军师团”作为第一个模板 |
| 角色数量 | 第一轮最少几个角色发言，是否必须包含主持人 |
| LLM 接入 | 当前使用真实模型还是 mock 优先 |
| 输出长度 | 第一轮每个角色发言字数限制是多少 |
| 失败策略 | 模型失败后展示错误、重试，还是直接使用 mock |
| UI 粒度 | 当前只做基础消息列表，还是引入简单气泡样式 |

---

## 8. 迭代完成定义

满足以下条件时，迭代 1 可视为完成：

1. 用户能输入议题并启动讨论。
2. 系统能自动加载默认模板。
3. 页面能展示用户消息、主持人消息和至少 2 个角色消息。
4. LLM 调用通过统一 client 完成。
5. Prompt 构造逻辑不在页面组件中。
6. 模型失败不会导致页面崩溃。
7. 工程构建和 Lint 通过。
