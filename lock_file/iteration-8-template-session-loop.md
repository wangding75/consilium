# 迭代 8：模板扩展与会话闭环

## 1. 迭代目标

在已有完整讨论能力和主界面基础上，将产品从“单模板单次体验”扩展为“多模板、多会话、可恢复”的产品闭环。

本迭代仍不做复杂配置 UI，但要完成模板数据抽象和会话存储闭环：

```text
多个内置模板 -> 新建讨论 -> 保存会话 -> 历史会话 -> 继续讨论
```

这一步完成后，产品具备基础可用形态，可以进入后续产品化阶段。

---

### 2.1 本迭代包含

| 范围 | 说明 |

## 3. 产品需求

### PRD-8.1：支持多个内置模板

**需求说明**  
作为用户，我希望可以选择不同讨论场景，而不是只能使用一个默认模板。

| 编号 | 需求 |
|---|---|
| PRD-8.1.1 | 系统至少内置 2-3 个模板 |
| PRD-8.1.2 | 每个模板有名称、描述、主题、角色列表 |
| PRD-8.1.3 | 用户可以从简单模板列表中选择一个模板 |
| PRD-8.1.4 | 新建讨论时使用用户选择的模板 |
| PRD-8.1.5 | 模板选择不要求复杂 UI，但必须可用 |

建议内置模板：

| 模板 | 说明 |
|---|---|
| 三国军师团 | 国风策略讨论场景 |
| 创业公司董事会 | 商业决策讨论场景 |
| 通用决策圆桌 | 泛用问题分析场景 |

---

### PRD-8.2：用户可以新建讨论

**需求说明**  
作为用户，我需要可以开始一次全新的讨论，避免旧会话状态干扰新议题。

| 编号 | 需求 |
|---|---|
| PRD-8.2.1 | 用户可以点击新建讨论 |
| PRD-8.2.2 | 新建讨论时需要选择或使用默认模板 |
| PRD-8.2.3 | 新讨论应生成新的 sessionId |
| PRD-8.2.4 | 新讨论不应污染旧会话消息 |
| PRD-8.2.5 | 新建后用户可以输入新的议题 |

---

### PRD-8.3：会话可以自动保存

**需求说明**  
作为用户，我希望讨论内容能保存在本地，刷新页面或下次打开后仍可查看。

| 编号 | 需求 |
|---|---|
| PRD-8.3.1 | 讨论会话自动保存到本地 IndexedDB |
| PRD-8.3.2 | 保存内容包括 topic、templateId、phase、messages、updatedAt |
| PRD-8.3.3 | 页面刷新后可以恢复最近一次会话 |
| PRD-8.3.4 | 保存失败时不影响当前讨论继续进行 |

---

### PRD-8.4：用户可以查看和恢复历史会话

**需求说明**  
作为用户，我希望能查看之前讨论过的问题，并继续某个历史会话。

| 编号 | 需求 |
|---|---|
| PRD-8.4.1 | 提供本地历史会话列表 |
| PRD-8.4.2 | 会话列表显示议题、模板名、更新时间 |
| PRD-8.4.3 | 用户点击历史会话后恢复消息和讨论状态 |
| PRD-8.4.4 | 恢复后用户可以继续输入，让讨论继续 |
| PRD-8.4.5 | 用户可以删除不需要的历史会话 |

## 4. 技术需求

### TECH-8.1：模板注册表

建议目录：

```text
src/data/templates/
├── index.ts
├── sanguo.ts
├── boardroom.ts
└── generic.ts
```

| 编号 | 技术需求 |
|---|---|
| TECH-8.1.1 | 所有内置模板通过 `templates/index.ts` 统一导出 |
| TECH-8.1.2 | 模板必须符合统一 Template 类型 |
| TECH-8.1.3 | 模板 id 必须稳定，不能随意变更 |
| TECH-8.1.4 | 模板需要包含 version 字段 |
| TECH-8.1.5 | 模板数据不应写死在页面组件中 |

---

### TECH-8.2：Session 数据结构

建议定义：

```ts
export interface DiscussionSession {
  id: string
  templateId: string
  topic: string
  phase: Phase
  messages: Message[]
  engineState: EngineState
  createdAt: number
  updatedAt: number
  version: string
}
```

| 编号 | 技术需求 |
|---|---|
| TECH-8.2.1 | session id 创建后保持稳定 |
| TECH-8.2.2 | session 保存完整 messages |
| TECH-8.2.3 | session 保存 engineState，支持恢复讨论 |
| TECH-8.2.4 | session 保存 templateId，用于恢复角色和主题 |
| TECH-8.2.5 | session 包含 version，便于未来迁移 |

---

### TECH-8.3：IndexedDB 会话存储

建议完善：

```text
src/db/index.ts
src/db/sessions.ts
src/db/templates.ts
```

| 编号 | 技术需求 |
|---|---|
| TECH-8.3.1 | 使用 IndexedDB 保存本地会话 |
| TECH-8.3.2 | `sessions.ts` 提供 create、update、get、list、delete 接口 |
| TECH-8.3.3 | list 接口按 updatedAt 倒序返回 |
| TECH-8.3.4 | 保存操作应与页面逻辑解耦 |
| TECH-8.3.5 | 存储失败需要捕获错误，不能导致讨论中断 |

---

### TECH-8.4：Store 与持久化同步

| 编号 | 技术需求 |
|---|---|
| TECH-8.4.1 | chat store 支持 loadSession、saveSession、resetSession |
| TECH-8.4.2 | 新消息追加后触发会话保存 |
| TECH-8.4.3 | 新建讨论时清空当前会话状态并创建新 session |
| TECH-8.4.4 | 恢复历史会话时同步 messages、engineState、templateId |
| TECH-8.4.5 | 避免保存过程和 AI 生成过程互相阻塞 |

---

### TECH-8.5：模板选择的最小实现

当前不做复杂配置 UI，但需要可用的模板选择方式：

| 方案 | 要求 |
|---|---|
| 简单列表 | 显示模板名称和描述，点击选择 |
| 默认模板 | 未选择时使用默认模板 |
| 新建讨论入口 | 新建时可选择模板或沿用当前模板 |

| 编号 | 技术需求 |
|---|---|
| TECH-8.5.1 | 模板列表来自模板注册表 |
| TECH-8.5.2 | 选择模板后写入当前会话状态 |
| TECH-8.5.3 | 不在当前迭代实现复杂筛选、导入、角色编辑 |

---

### TECH-8.6：数据迁移预留

| 编号 | 技术需求 |
|---|---|
| TECH-8.6.1 | IndexedDB 数据库版本明确 |
| TECH-8.6.2 | session 和 template 数据包含 version |
| TECH-8.6.3 | DB 初始化逻辑支持后续升级 object store |
| TECH-8.6.4 | 当前不要求实现复杂 migration，但不能阻碍后续扩展 |

## 7. 需求评审关注点

进入本迭代开发前，需求评审需要确认：

| 评审项 | 需要确认的问题 |
|---|---|
| 模板数量 | MVP 内置几个模板，分别是什么 |
| 模板质量 | 新模板是否需要完整角色 Prompt，还是先用简化版本 |
| 历史入口 | 历史会话列表放在侧边栏、首页，还是讨论页内 |
| 自动保存频率 | 每条消息保存、每轮保存，还是定时保存 |
| 恢复策略 | 默认恢复最近会话，还是让用户手动选择 |
| 删除确认 | 删除历史会话是否需要二次确认 |
| 本地数据风险 | 是否需要提示数据仅保存在本地浏览器 |

## 8. 迭代完成定义

满足以下条件时，迭代 8 可视为完成：

1. 产品支持多个内置模板。
2. 用户可以新建讨论、查看历史、恢复会话、继续讨论。
3. 会话数据可以保存到 IndexedDB。
4. 模板和会话数据结构具备版本字段。
5. 不依赖配置 UI 也能完成多场景体验。
6. 工程构建和 Lint 通过。

## 9. 后续衔接

迭代 8 完成后，产品具备 MVP 体验闭环。后续可进入产品化阶段：

| 后续方向 | 说明 |
|---|---|
| 模板动态下发 | 通过服务端接口拉取模板和版本 |
| 语义爽点检测 | 使用 Embedding 或 LLM Judge 升级事件判断 |
| 多模型适配 | 根据不同模型能力适配 Prompt |
| 配置 UI | 补充模板选择、LLM 配置、角色配置页面 |
| 移动端适配 | 提升移动端讨论体验 |
| 监控与埋点 | 分析完成率、继续率、用户主动发言率 |
